關於權限設定

2017-02-22

看到一篇不錯的權限說明

“2775” is an octal number that defines the file permissions. The leftmost digit (the “2”) is optional and defaults to zero if not specified.

The digits in the “775” portion define the permissions for the file owner, file group, and everyone, from left to right respectively. Valid numbers are 0-7, with 4 indicating read access, 2 indicating write access, and 1 indicating execute access (for directories execute permission also allows you to traverse the directory tree). If a file has multiple permissions, you add the values together. Thus a “7” means 4+2+1 which is read, write, and execute permission, while a “5” means 4+1 which is read and execute permission.

More information about these permissions can be found here.

The first digit of a 4-character permission (in this case, the “2”) specifies the setuid mode (=4), the setgid mode (=2), and the sticky mode (=1). These values are added together in a manner identical to the read/write/executable permissions above. More information is here.

It should be noted that the setgid mode and the sticky mode have special meanings for directories.

Thus a permission of “2755” for a directory means that everyone has read and execute permission, while the file owner and members of the file’s group additionally have write permission. And any files or subdirectories created in that directory will inherit the parent directory’s group id.

一週寫出第一個Android App

2016-07-11

前言

暑假接下Dean’s Challenge，目前在T&O(Training & Orientation)階段發想暑假時大學生會碰到的問題。我們將目標設定在解決暑假太熱的問題，但經過訪談後發現大部分人覺得流汗造成的濕黏不適感才是主要的困擾。

我們的Insights分為外在與內在部分：

  • 流汗的肌膚與其他物體接觸會帶來不適感 // 例如手被汗黏在桌子上
  • 在走動時會產生風，也比較不會注意身上的汗，不適感較少；停在路上（例如等紅燈）、熱消耗時卻會注意到身體的濕黏不適
    在經過討論後，我們決定從心理、轉移注意出發，設計出能夠轉移「停在戶外時產生濕黏」注意的Solution，並將使用者設為台大的學生。

產品

我們構想出在紅綠燈柱貼上QR Code以及吸引人去掃的Icon與指示，使用者在等紅燈時透過我們的App掃QR Code，就可以蒐集到台大各系的學生角色，與在台大發生的不同事件。藉由這些事件可以培育你的角色，從大一開始，最後到大四並成功畢業。

事件分為四大類：

  • 升級：畢業的必要過程，例如大一結業升到大二
  • 年級特有：不同年級時發生的事件，例如大三專題
  • 學系特有：不分年級、各系特有的事件，例如舞會
  • 不分系：台大共有的事件，例如音樂節

因此，撰寫App（而且只有一週QQ）就是技術部的工作了。

Day 1 將環境裝好

因為之前就裝過Java的JDK，因此安裝Android的SDK比較容易。但因為不喜歡Eclipse的介面，於是決定用Android Studio進行開發。

Day 2 QR Code 掃瞄器

選用Zxing，因為我們只有要使用QR Code而無其他條碼，而且掃描的速度對我們暫時不那麼重要，因此不選用ZBar。之後照著Github的說明，就完成QR Code掃瞄器的設定。這邊大致要加上的設定只有Android Manifest的xml檔要允許相機的權限（uses permission, CAMERA）以及在將來測試的AVD設定裡允許取用裝置的相機，例如我用mac測試，就允許他相機取用mac的webcam0。

Day 3 排版

就像網頁一樣，因為只是要做簡單的demo，因此我們在設定物件的位置時常用android:layoutMargin去設定margin值，並且將數字寫死，單位使用dp。在未來可能需要更好的寫法。此外置底的四個功能按鈕使用了android linear layout，設定orientation為horizontal，並把每個權重（gravity）設為0.25（每個加起來需等於1）。

Day 4 轉頁

在撰寫時，例如從here轉頁到there後，若there已結束要返回here，本來是開一個新的intent去轉頁，但後來發現使用finish()方法更有效率。

Day 5 Dialog

本來在遇到角色與事件時都使用AlertDialog製作跳出視窗，但因為我們的跳出視窗中包含整個角色或事件的圖片，首先使用了setContentView()將圖片與其上的按鈕另開一個Activity，並將該Activity包入AlertDialog，後來因為無法取得content中設定按鈕的onclick method，因此demo時改用轉頁處理了對話（dialog）的問題。

Day 6 輸出apk

輸出apk時遇到了一些大問題，例如Android Studio說我們的app有太多的methods（但其實很少）、too much work on main thread等，Android允許64K的使用量，我們的app卻用到超過因此無法。上網查可使用Multidex（只是我裝好依然失敗），一個簡單的app就這麼完成了。

HCD Week2 訪談一

偉立 21歲大學生 排球少年

家裡的飲料只有青草茶、牛奶（和減糖豆漿），所以早餐配飲料通常都從中挑選。
不特別愛喝手搖杯，大約平均一週喝一次。
在家是媽媽煮菜，非常均衡而且好吃。肉的量偏多，因為哥哥愛吃。
最近在學校常去宿舍吃自助餐，再來為求方便會去吃小福的八方雲集或麻辣燙。
自己不會料理，交給媽媽打理。前陣子有在料理，但不是為了健康或生活需求，單純只是嘗試食譜。
喜歡吃的東西：不太甜的巧克力。
自認最近飲食還算健康。自覺飲食不健康的時候通常以運動彌補，較少透過飲食解決。
覺得營養均衡、五類都有攝取到就算是健康，也不能狂嗑某種食物。
家裡的食材通常在全聯購買，騎腳踏車可到。傳統市場太遠。有去過傳統市場但平常不會去。
假如下一餐要吃得健康，在外會去吃自助餐或便當這種可以自己選擇菜色的地方，不然會自己去市場、超市買食材全部丟進去煮成火鍋。
平常的身份、職業（學生）會影響飲食，因為現在是給別人養。以後工作就會隨便亂吃。
系隊的身份也會影響飲食，下禮拜要比賽不能喝飲料。
想到健康飲食的相關機構：有機商店，覺得不合理：「他省了農藥的錢還賣比較貴」。

HCD week 2

2016-05-13

Human-Center Design Week2 Summary

因為HCD課程提供的PDF都是全英文，決定從本週開始一邊閱讀一邊留下筆記，也幫助自己未來review能更有效率。每週17頁全英文PDF不得不說還是有滿大的閱讀壓力啊。QQ

以下正文開始
在這個階段最重要的是了解設計對象的需求。

Step1 在團隊討論，搜集想法的過程中，可以先整理已掌握的背景知識以及可能用到但尚未了解的領域知識。在開始集思廣益之前，也要對case本身的限制有所瞭解。這些都是有助你整理思考方向，不會討論得太廣無法收斂，也不會發生想法太過單一、沒有空間發展的情況。

Step2 Do some research.這裡提供4種方式認識瞭解設計對象的需求：

A.Learn from people
(案例：在肯亞首都奈洛比，從當地市場蔬果商獲得資訊，以提升都市食物安全)
每個受訪者都是關於他們生活的專家。因此保持好奇並給予應有的尊重。

Define your audience
先想好誰是你設計思考的對象，他們以及其生活週遭的人們都是你的訪談對象。

  • 符合大眾的一般使用者與極端用戶都應被訪談。極端用戶能幫助你想到不曾想過的使用情境，並且大多情況下他們也符合一般用戶的使用模式。
  • 先想好要如何進行訪談。如何幫助你更瞭解設計的對象？
  • 別畏懼尋求認識的人脈關係，人們通常樂於分享自身經驗。
  • 保持訪談過程舒適，可以輕鬆、不相干的閒聊開始訪談，並適時關注受訪者是否覺得不適。注意保持適當隱私。

B.Learn from experts
在從專家（案例：菜市場的蔬果攤）身上獲得該領域知識的同時別忘了也要保留空間，跳脫某領域的既定模型框架，才能生出好的設計。

  • 隨時做好準備，以免訪談內容和預期不同。
  • 可請專家協助形塑計劃前期的概念。
  • 善用通訊軟體如Skype，可以即時呈現視覺成果。

C.Immerse yourself

  • （案例：既然要設計一款新的鍋爐，不如花一天親自下廚）
  • There’s no better way to understand the people you’re designing for than by immersing yourself in their lives and communities.
  • Customer Journey: 觀察案例從頭到尾與自身經歷的差別。這些差別帶來什麼感覺？有哪些地方是沒有預期到的、有挑戰性的、Seamless無間斷一直在發生的？
  • 在觀察當下就紀錄下來，作為最寫實、準確的紀錄。

D.Seek analogous inspiration

  • （案例：觀察賽車維修團隊的分工合作）
  • 吸～收～學～習～將所觀察到的內化，用在自己的project上。
  • Brainstorm Analogous Experiences 在紀錄下的觀察旁寫下是在何種狀況下觀察到的。
  • 先想著從對象學習，dont worry about making sense.

Step3 Think into it. Stay on track.

  • 紀錄問題時盡可能簡單，可以增加與受訪者眼神接觸的時間。
  • Open general, and go deep.
  • Conversation starters：用簡單的模型、情境等描述來分享project的初步架構。
  • 比起用自己的話記下受訪者的意思，不如直接quote他說的話。在與team討論時能夠獲得更多啟發。
  • Visuals often prompt more conversation.鼓勵受訪者用畫圖等方式描述他想表達的東西。

Step 4 其他的訪談方法

  • Personal Diary（訪問個人經驗）
  • Photo Essays（利用受訪者所拍攝的照片瞭解其人）
  • Customer Journey（紀錄一個經驗的timeline，瞭解其各階段的感受）
  • Card Sorts（從受訪者對物件的重要性排序瞭解其決策方式）
  • Concept Provocations（以繪畫與文字解釋概念）

Step 5
觀察到馬上紀錄，並花時間消化。
